## 推論穩定性改善方向

### 告警通知

- Telegram Bot 是目前最可行的選項（Line Notify 已死）
- 推論成功 / 失敗都要通知，不只是失敗
- 考慮用 GitHub Actions 做監控？每天 18:00 去檢查 GitHub 上的結果日期，超過 1 天就 alert

### WSL2 冷啟動問題

- 延遲啟動？讓 Task Scheduler 等 WSL2 準備好
- 或是把推論腳本包成一個 .sh，裡面先 sleep 幾分鐘再開始跑
- 更根本的解法：把推論移到雲端（但費用問題）

### git push 失敗重試

- 現在是 non-fatal，改成重試 3 次，還是失敗才發告警
- 或是用 GitHub Actions 來主動 pull 結果，而不是 push？

## 模型改善方向

### Scale Gate 均衡化

- 觀察 V6.2 padding mask 後的效果
- 如果還是偏 Long：
  - Branch-level dropout（訓練時隨機 mask 某個 branch）
  - Loss 正則化（讓 gate 不要太極端）
  - 直接固定 gate 比例（但這樣就不是自適應了）

### RS 特徵（V6.2 訓練用）

- RS_5d / RS_20d / RS_60d — 相對強度

### Walk-Forward 驗證

- 目前只有 Quick Fold IC=0.0744，不夠嚴格
- 真正需要做：時間序列 walk-forward，確保 out-of-sample 表現一致
