# Phase 4 設計筆記：產業理解融合（KG 豐富化 + 產業變化 Agent）

> 建立：2026-06-27　狀態：**計畫定稿、待 Phase 3 完成後接續**（尚未動工）
> 性質：策略路線設計，非實作。實作前仍須依互動規則 2 列計畫、確認後再動。

---

## 0. 一句話定位

把「產業／供應鏈的理解」融進 MarketMamba。核心認知：**現有模型只從數字做橫斷面預測，對產業敘事的理解是間接的**。本計畫不是把敘事硬塞進模型，而是**並行加一層**——符合既有隔離原則（V6.1 是紅線、新東西一律附加並行）。

---

## 1. 動機脈絡（Thread 討論）

來源是 Threads 上關於「股癌為何資訊系統快」的討論，提煉出兩個可操作內核：

- **方法 1：先建地圖、再追變化** — 重點不是看新聞，是建立產業上下游關係圖，再監控「規律出現變化」（法說會語氣、上游報價、下游 capex、庫存）。
- **方法 2：玩 product cycle + catalyst**（第二位網友）— tech investor 靠 channel check 維持 12–18 個月能見度，等法說會/新聞出來都是舊訊息。

**誠實校準**：第一篇有招生成分，論述被包裝得比可操作性漂亮。真正可落地的只有「建地圖」和「追變化」兩條；「領先半年看 design win／catalyst 預判」需要一手 channel check 與付費資料，自動化系統做不到，硬做＝幻覺產生器（也正是文章自己警告的「更吵的新聞機器」）。

---

## 2. 可行性調查結果：櫃買「產業價值鏈資訊平台」（ic.tpex.org.tw）

調查日 2026-06-27，實際進站確認。

**技術面**：每條鏈是固定網址 `introduce.php?ic=XXXX`（如 D000=半導體），server 直接吐 HTML 表格，**不需登入、非純前端渲染、好爬、低維護**。整站約 30+ 條鏈（半導體／生技醫療／綠能三大類 + 近年加的雲端、AI、自動化、太空衛星、被動元件），一次抓完約 30 幾頁，季度刷新即可。

**抓得到（核心價值）**：
- **節點 → 公司對應**：每條鏈分成多個功能節點（半導體鏈有 28 個：IC設計、電源管理IC、記憶體、晶圓代工、封測…），節點內公司分「本國上市／上櫃／興櫃／外國上市／知名外國企業」。例：IC設計節點＝凌陽、智原、創意、全訊、晶心科、芯鼎、世芯-KY…
- **節點有順序**（上游→中游→下游，依序排列）→ 可推出位置上的方向性。
- **每條鏈的敘述文字 + 政府相關產業政策**（櫃買編輯式撰寫，內文已寫到 2025下半年/2026上半年，代表**有更新但屬季度級、非即時**）。

**抓不到（誠實缺口）**：
- **沒有股票代號**，只有公司「名稱」→ 需 name→stock_id 對應；KY/外國不在台股 universe。
- **沒有任何「變化訊號」**：法說會逐字稿、月營收、三大法人、庫存、capex 一個都沒有。它是靜態結構，不是動態敘事流。
- 公司自填的產品介紹／財報連結品質參差、雜訊高。

**定位結論**：ic.tpex 幾乎完美解決「方法1：建地圖」，但對「方法2：追變化」**零貢獻**。它是很好的第一塊磚、不是整面牆；變化偵測的資料源在別處（見 §4）。

## 3. Phase 4-A：知識圖譜（KG）豐富化 — 進模型

**目標**：用產業鏈資料給 GATv2 加上有語意標籤的邊（現有邊大多是統計相關性算出來的）。這是唯一能真正進到模型、又完全不碰 production 推論的接點，重訓時生效，CP 值最高。

**邊的設計（已定：要做方向性）**：
- 同節點公司之間 → **競品邊**（無向）
- 相鄰節點之間（上游→下游）→ **有向上下游邊**

**⚠️ 方向的本質（重要限制，不可假精確）**：
ic.tpex 給的是**「位置上的方向」（A 節點在 B 節點上游），不是「驗證過的公司對公司供應關係」（A 公司實際賣給 B 公司）**。真正的廠商對廠商供應鏈要靠財報「主要客戶／供應商揭露」或付費資料（Bloomberg SPLC 那類）。
→ 因此 4-A 只建「鏈內上下游」的**粗粒度有向邊**，不假裝是精確供應鏈。已比現狀（純統計相關性）強，且不引入假精確。

**步驟**：
1. 爬 ic.tpex 30+ 鏈 → 公司×節點對應表（含節點順序）
2. name→stock_id 對應
3. 建邊（同節點競品 / 相鄰節點有向上下游）
4. 併入既有 KG cache（`Data/cache_v6/knowledge_graph_cache.npz`）
5. 下次重訓驗證 IC 是否提升（對照無此邊的基準）

**更新頻率**：季度級（KG 邊本來就該慢更新，反而與平台更新節奏匹配）。

## 4. Phase 4-B：產業變化 Agent — 獨立層，不碰 production

**目標**：實現「方法2：追變化」。本質是 LLM 工作、不是時序模型工作 → 留在獨立 Agent／敘事層（每週摘要），**不硬塞回 Mamba 當特徵**。完全附加、並行、失敗不影響線上，比照雙模型那套打法。

### 4.1 資料源地圖（已查證，2026-06-27）

好消息：**80% 用既有 FinMind token 免費可得，20% 要付費**。

| Thread 講的訊號 | 資料源 | 取得性 |
|---|---|---|
| 營收動能轉折、重複提到新應用 | FinMind `TaiwanStockMonthRevenue`（月營收 YoY/MoM surprise） | ✅ 已有，月更，訊號強 |
| 下游客戶 capex 增加 | FinMind 現金流量表（購置不動產廠房設備） | ✅ 已有，季更 |
| 庫存變化 | FinMind 資產負債表（存貨）→ 存貨周轉天數 | ✅ 已有，季更，領先訊號佳 |
| 機構連續買 | 三大法人 / 融資券 | ✅ 系統已是特徵 |
| 法說會語氣差異（這次 vs 上季） | MOPS 法人說明會簡報 PDF（`t100sb07_1`，依股號下載） | ✅ 免費下載，需 LLM 做語氣/關鍵字 diff |
| 重大訊息／公告 | MOPS 重大訊息 | ✅ 免費，每日 |
| **上游報價轉折（catalyst／高頻報價）** | 集邦 TrendForce / DRAMeXchange / WitsView | ❌ **付費，本期跳過（見決策）** |

### 4.2 兩層如何合起來（真正價值）

**第一層的鏈分組 = 第二層訊號的歸納框架。** 單看一家公司月營收跳升是噪音；但若偵測到「某條鏈的中游存貨下降 + 下游 capex 上升 + 整鏈月營收同向」**整條鏈節點同向移動**，就是 thread 的「規律出現變化／供應鏈大洗牌」訊號——這才值得提醒。
→ ic.tpex 給鏈結構，FinMind/MOPS 給各節點變化，LLM 負責比對整條鏈是否同向表態。

### 4.3 範圍（已定）

不特別訂 MVP，目標是看到**完整最終效果**：
- 守備清單 = ic.tpex 鏈分組
- 訊號源 = FinMind（月營收/存貨/capex/法人）+ MOPS（法說會PDF/重大訊息）
- LLM 做「本季 vs 上季」語氣/關鍵字 diff
- 輸出每週「整條鏈同向變化」摘要，只在偏移夠大時提醒
- 沿用既有 `llm/report_generator.py` 為基礎，但走獨立 pipeline

## 5. 決策紀錄

- **4-A name→stock_id 對不到的公司：保留標記、不連邊、需記錄（2026-06-27）**：KY、外國企業、改名/別名對不到 stock_id 者，**保留在對應表中並加標記**（標明原因：foreign / KY / unmatched），但**不建立邊**進 KG。需可被查詢與後續補回，不可直接丟棄。
- **付費催化劑資料（集邦報價等）：本期跳過、但需記錄「有此資料、因付費略過」（2026-06-27）**：第二位網友的「高頻報價＝catalyst」是唯一要錢的部分。**目前不做**，但要在計畫/資料源清單中明確記錄「該資料源存在、能補上報價轉折訊號、因需付費而暫緩」，待系統真的有獲利後再評估接入。不是「做不到」，是「成本決定先不做」。
- **方向性邊採位置粒度、不假裝是精確供應鏈（2026-06-27）**：見 §3。只建鏈內上下游粗粒度有向邊；精確廠商對廠商供應關係需財報揭露或付費資料，不在本期範圍。
- **變化訊號留 Agent 層、不進模型特徵（2026-06-27）**：變化偵測是 LLM 工作。少數（月營收 surprise、存貨周轉）未來重訓時「可考慮」數值化進特徵，但屬大工程，列 Phase 4 之後再評估，本期不做。
