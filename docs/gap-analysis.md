# MarketMamba 台股分析平台 Gap Analysis

> 專案目標已確認為**台灣股市分析**，與加密貨幣無關。以下報告整合前端盤點結果、金融專家台股分析框架與 UI 重組建議，聚焦於：現況、差距、風險與可執行改善方向。

## 2. 已發現的 6 大技術問題清單

> 原始盤點中已確認多項風險，以下整合為 6 大問題類型，便於後續排程修復。

| # | 技術問題 | 影響範圍 | 風險 |
|---|---|---|---|
| 1 | **假資料混入正式畫面** | InvestmentSim、Portfolio | 讓使用者誤以為績效／損益為真實結果，破壞可信度 |
| 2 | **硬編碼指標與模型資訊** | QuantAnalysis、ModelStatus | 畫面看似完整，但實際無法反映最新資料 |
| 3 | **API 路徑/模組風格不一致** | reports.js、各 API 模組 | 可能造成 `/api/api/...` 重複、維護困難、除錯成本高 |
| 4 | **缺失值防護不足** | StockModal、SectorHeatmap 等 | `undefined.toFixed()`、NaN 顯示、元件崩潰風險 |
| 5 | **元件缺少 loading/error/empty 狀態** | SectorHeatmap、部分頁面 | 真實 API 接入後 UX 不穩定，錯誤時無法回退 |
| 6 | **前端展示與後端資料契約未明確** | 多數頁面與 API 模組 | 容易導致欄位名稱漂移、假資料長期留存、功能無法落地 |

### 2.1 問題說明補充

#### 1) 假資料混入正式畫面
- `InvestmentSim` 的模擬報酬包含 `Math.random`
- `Portfolio` 的損益曲線為前端生成
- 這類做法在 demo 階段可接受，但若進入正式產品會直接影響信任

#### 2) 硬編碼內容過多
- `QuantAnalysis` 大量技術／籌碼數據是靜態常數
- `ModelStatus` 的 badge、checkpoint 資訊部分為寫死
- 容易產生「畫面很完整、資料卻不真實」的落差

#### 3) API 不一致
- `reports.js` import 風格與路徑前綴可能不一致
- 若 `client.js` 已設 `baseURL=/api`，模組內再手動補 `/api/...` 可能產生重複前綴

#### 4) 缺失值處理不足
- `StockModal` 多處 `toFixed` 直接使用
- `SectorHeatmap` 在 `alpha_20d` 缺失時可能出現 NaN
- 台股資料來源常有缺欄位、停牌、尚未公告等情境，必須做 robust handling

#### 5) 狀態處理不足
- `SectorHeatmap` 無 loading / error / empty state
- 某些頁面雖有 SkeletonLoader，但未全面統一導入
- 真實 API 在免費方案／學生開發環境下更容易有 timeout 或 quota 問題，因此狀態處理更重要

#### 6) 資料契約不明確
- 現有 API 模組雖已拆分，但欄位契約、回傳格式、空值規則未見統一文件
- 這會讓前後端難以並行，也會讓 mock 與 production schema 漸行漸遠

## 3. 頁面重組對照表（現有 7 頁 → 建議 5 區）

> 設計專家建議將目前分散頁面重構為更貼近投資流程的資訊架構。

| 現有頁面 | 路由 | 主要角色 | 建議併入的 5 大核心區 | 重組說明 |
|---|---|---|---|---|
| Dashboard | `/` | 大盤總覽、關鍵指標卡 | **1. 市場總覽** | 保留為首頁核心 |
| MarketView | `/market` | 市場資料、AI 日報 | **1. 市場總覽** / **5. 消息與日報** | 可拆成「總覽」與「日報/事件」兩塊 |
| TradingSignals | `/scanner` | 掃描器訊號 | **2. 選股 / 訊號中心** | 作為選股入口與策略雷達 |
| QuantAnalysis | `/quant` | 技術/籌碼/量化摘要 | **3. 個股分析中心** | 不應只是獨立靜態頁，應成為個股頁的分析 Tab |
| Portfolio | `/portfolio` | 個人持股與損益 | **4. 投資組合 / 策略追蹤** | 保留，但需真實化 |
| ModelStatus | `/model` | 模型健康度、績效 | **4. 投資組合 / 策略追蹤** | 可下沉為策略後台，不必佔主導頁面 |
| InvestmentSim | `/sim` | 模擬績效、再平衡 | **4. 投資組合 / 策略追蹤** | 與模型/投組整合成單一策略模組 |

### 3.1 建議 5 大核心區

1. **市場總覽**  
   - 大盤、指數、風險情緒、AI 日報、族群輪動
2. **選股 / 訊號中心**  
   - 掃描器、多因子條件、熱門訊號、候選名單
3. **個股分析中心**  
   - 個股頁（技術、籌碼、基本面、事件、估值）
4. **投資組合 / 策略追蹤**  
   - 持股、績效、模擬、再平衡、模型監控
5. **消息與日報**  
   - 市場日報、個股重大訊息、法說、除權息、政策/產業新聞

### 3.2 個股頁建議定位

目前最大的結構缺口是**缺乏個股分析中心**。根據設計規劃，個股頁應成為平台主軸，並可採 8 Tab 架構，例如：
- 總覽
- 技術面
- 籌碼面
- 基本面
- 估值
- 消息事件
- 同業比較
- 策略/回測

## 4. 缺失的關鍵功能

以下是從「台股分析平台」定位出發，目前最需要補齊的功能模組。

### 4.1 個股分析頁（最高優先）

**現況：**
- 只有 `StockModal` 作為局部資訊容器
- 缺少完整個股頁與深度分析流程

**缺失內容：**
- 個股總覽（價格、成交量、漲跌幅、產業、排名）
- 技術面（均線、RSI、MACD、型態、支撐壓力）
- 籌碼面（三大法人、融資融券、當沖比、券資比）
- 基本面（營收、EPS、ROE、毛利率、估值）
- 事件面（法說、除權息、重大訊息、處置）
- 同業比較 / 族群定位

### 4.2 籌碼面模組（台股差異化關鍵）

**現況：**
- 僅在 QuantAnalysis 中零散存在
- 且疑似有硬編碼資料

**建議補齊：**
- 三大法人買賣超與連續天數
- 融資融券餘額與變化率
- 券資比、融資使用率
- 當沖比、週轉率、籌碼集中度
- 期貨/選擇權情緒代理指標

### 4.3 基本面模組

**現況：**
- 沒有完整的基本面頁或資料結構

**建議補齊：**
- 月營收 YoY / MoM
- EPS、ROE、毛利率、營益率
- 現金股利、殖利率
- 本益比 / PB / 估值區間
- 近期財報摘要與趨勢圖

### 4.4 消息 / 事件中心

**現況：**
- 只有 MarketView 的日報雛形

**建議
