# 資料/特徵基礎升級計畫 — 把 baseline_common 扶正成下一代基礎

> 日期：2026-07-14
> 緣起：Cowork 討論——原本考慮「開一個全新平行專案、只共用前端」重新蓋一遍 pipeline，最後決定改成**擴充 `V6/experimental/baseline_common.py` 已經驗證過的資料層**，不重寫整個系統。V6.1 全程不動，符合既有隔離原則。
> 一句話定位：方向二為了公平比較四個模型，被迫蓋出一套比 V6.1 現有管線更嚴謹的資料層（協定凍結、D0–D5 驗證方法論）。與其丟掉重寫，不如把這套已經驗證過的東西正式扶正成下一代基礎，Mamba 變成這個梯子上「多一個模型」，而不是另起一個平行宇宙。

---

## 0. 決策脈絡（為什麼不開全新專案）

你想補的「洞」，仔細看大多是**資料層**問題（存活者偏差、Close=0 損壞列、還原/未還原價接縫、point-in-time 正確性），不是模型層問題。不管包在哪個新專案裡，只要底層還是同一批 FinMind/yfinance/TWSE 資料，問題就還在。真正能解決這些洞的是「用協定驅動、逐步驗證的方法論重蓋資料層」，而這件事已經在方向二裡做出雛形了。另一方面，fetcher（含重試/欄位對映修復史）、KG 建構、訊號掃描器、LLM 報告產生器，都是踩過真實坑才長出來的資產，重寫等於冒著重新踩坑的風險換一個「看起來比較乾淨」的感覺，這個交換在你现在的時間預算下不划算。

---

## 1. 範圍界定

### 保留不動（沿用既有、已驗證過的工程資產，不重寫）

- 資料抓取 `fetcher.py`（FinMind + yfinance + TWSE/TPEX direct，含重試與 07-07 修復的欄位對映邏輯）
- 知識圖譜建構 `graph_builder.py`（含未來 Phase 4-A 產業鏈擴充計畫）
- 訊號掃描器 / 型態學 / 四層退場邏輯（`signals/`、`quant/pattern_scanner.py`）
- LLM 市場報告產生器 `llm/report_generator.py`
- 前端（`app/frontend`、`PersonalOS`）與現有後端 API 契約
- **V6.1（`V6/models/`、正式推論線）本身**——這是紅線，本計畫任何階段都不動它

### 升級目標：`baseline_common.py` → 正式的下一代資料/特徵基礎

現有已具備、可以直接繼承的部分：universe 過濾協定（§2）、point-in-time 驗證慣例（D5 已驗證無位移）、`clean_and_scale` 標準化、D0–D5 那套「先懷疑、再驗證」的方法論本身。

需要新增才能扶正成正式基礎的部分：

1. **序列張量輸出模式**：目前 `baseline_common` 只吐扁平 300 維（給 Ridge/GBDT/LSTM），要新增 `(N, 252, 59)` 序列輸出模式，才能餵給 Mamba+GATv2
2. **KG 邊整合點**：讓新基礎能對接 GATv2 讀取知識圖譜邊，不是只服務扁平模型
3. **修掉本次診斷發現的三個資料衛生問題**：Close=0 損壞列、還原/未還原價接縫、超限報酬複驗——直接在新基礎版本修掉，不動 V6.1 現有 pipeline
4. **協定文件正式版本化**：比照 `baseline-experiment-protocol-draft-2026-07-11.md` 的凍結模式，把新基礎的資料契約寫成可長期參照的規格文件
