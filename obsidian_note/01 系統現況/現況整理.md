## 🎯 本階段主軸：系統穩定性（inference 不穩定）

依「Agent 分析結果」指出的優先順序，目前先處理**每日 17:00 WSL2 推論偶爾卡死/超時**（約每週一次）的問題。本階段已完成一輪強化：

1. **WSL2 暖機檢查**（`_wsl_warmup()`）：主推論前先跑 `wsl -d Ubuntu -e echo ok`（30 秒逾時），沒反應就跳過本次推論，不浪費 60 分鐘
2. **逾時自動重啟＋重試**（`_wsl_shutdown()`）：第一次 60 分鐘逾時 → `wsl --shutdown` 重啟 VM → 等 10 秒 → 再跑一次
3. **Telegram 後台告警（SYS-07 完成）**：失敗時連同 Windows Toast 一起發送，含各任務狀態/耗時/錯誤摘要/是否觸發過 wsl 重試
4. **自動化頁面重新設計**（`AdminOps.jsx`）：階段卡片清單（⚪/⚙️/✅/❌）取代單條 progress bar，原始輸出收合並過濾雜訊
5. **git push 失敗重試（SYS-08 完成）**：`_push_to_github()` 的 `git push origin main` 失敗自動重試最多 3 次（間隔 10 秒），逐次印出結果

**待觀察**：6/10、6/2 兩次嚴重卡死（60 分鐘超時、完全無輸出）根本原因仍未確認，懷疑與「電腦睡眠喚醒後 WSL2 VM 冷啟動卡住」有關（當天使用者不在家、電腦無人操作）。暖機+重試是否能避開這類情況，需等下次發生時驗證。

通知分工：**Telegram = 後台自動化告警**、**PWA 推播 = 前端一般通知**（PWA 使用者表示暫不需要，已擱置）。

## 最近完成的事（2026-06-07 ~ 2026-06-09）

### 推論管線關鍵 Bug 修復

**Scanner 機器人沉默失敗（`timedelta` NameError）**

- **根本原因**：`run_daily_inference.py` 的 `_archive_scanner_result()` 函式內使用 `timedelta(days=90)` 但沒有 local import，而 `timedelta` 只在 `main()` 的 `else` branch import，`_archive_scanner_result()` 呼叫時直接 `NameError`
- **影響範圍**：Step 7 的整個 `try` block 被 `except` 吃掉 → Alpha 機器人 + Scanner 機器人**連續失敗超過 14 個交易日**（`sim_backtest.json` 停在 2026-05-26）
- **修復**：在 `_archive_scanner_result()` 函式頂部加 `from datetime import timedelta`
- **恢復**：需刪除 `V6/results/sim_backtest.json` → 下次推論觸發全量重建，補回 5/27–6/08 的缺失紀錄

## PersonalOS 整合現況

MarketMamba 的功能同時在兩個前端跑，詳見 [[03 架構筆記/前端網頁]]。

**PersonalOS 比 Web 版多的功能**：

1. **後台管理（AdminOps）**：可以直接在 UI 裡觸發推論，不用打開終端機
   - 「純模型推論」：單獨補跑，即時進度條 + log 輸出（解析 `[N/10]` log 格式）
   - 「完整每日自動化」：補跑整天，含持倉同步 + 新聞，等同 `run_daily.bat`
   - 鎖檔偵測（PersonalOS 被關閉但 WSL 程序還在跑的情況）
   - 感覺陽春但其實很實用，主要用途是推論失敗時方便補跑

2. **多帳戶持倉**：永豐（Shioaji）+ 玉山（esun_trade），兩個帳戶合一介面
   - 點擊持倉列 → 退場條件 Modal（4 條件逐一顯示 ✅/❌）
   - 風險分數 0–100（exitSignal + 虧損幅度 + model_signal 綜合計算）
   - 注意：目前這個退場 Modal 只有 4 個基本條件，**不是完整的四層退場邏輯**，PR 3 才會真正接入 `signal_conditions.py`
